## Week 2 — Workflow Orchestration with Kestra

In Week 2 of the Data Engineering Zoomcamp, we continued advancing our automation workflow by introducing **workflow orchestration** using Kestra.

Kestra is an open-source orchestration platform designed to build scalable workflows using **code (YAML)**, **no-code interfaces**, or an **AI Copilot**. Orchestration helps streamline the management of multiple platforms, services, and applications that make up modern data pipelines.

Instead of manually executing steps, orchestration allows workflows to coordinate tasks such as:

* Extracting and transforming data
* Creating outputs
* Running queries
* Handling failures and retries
* Scheduling automated events

This enables fully automated end-to-end ETL pipelines.

---

## Understanding Kestra Workflows

Workflows in Kestra can be created using YAML definitions or through a no-code AI-Copilot interface. These workflows define each step required to orchestrate the ETL process from beginning to end.

The fundamental unit of orchestration in Kestra is called a **Flow**.

A Flow defines:

* Tasks
* Execution order
* Inputs and outputs
* Orchestration logic

### Required Flow Components

Every flow must include:

* **identifier (id)**
* **namespace**
* **list of tasks**

### Optional Flow Components

Flows can also include additional configuration:

* inputs
* outputs
* variables
* triggers
* labels
* pluginDefaults
* errors
* finally
* retries
* sla
* concurrency
* descriptions
* disabled
* revision
* checks

These options allow workflows to scale from simple automation scripts to production-grade orchestration systems.

## Extending Our Data Pipeline

During this week, we expanded our existing Docker setup by adding the **Kestra service** to our Docker YAML configuration.

We then used Kestra to:

* Load data files
* Create infrastructure using Infrastructure as Code (IaC)
* Store data in a GCP bucket
* Transform datasets using BigQuery

Before running workflows successfully, we needed to configure **GCP environment secrets** so Kestra could securely access cloud resources.

Kestra provides multiple methods for managing secrets. We configured them using the **Key Value (KV)** option and connected them within our workflow configuration.

We also implemented **Kestra trigger tasks** to:

* Schedule recurring jobs
* Backfill historical data when required

## Challenges Encountered

This week, I ran into an issue connecting GCP environment secrets to Kestra.

There are several approaches available:

* Using Key Value pairs inside Kestra
* Importing or extracting secrets directly from GCP

My configuration problem appeared to stem from choosing the Key Value Pair method and incorrectly updating secrets to environment variables.

Below is the configuration used:

```yaml
pluginDefaults:
  - type: io.kestra.plugin.gcp
    values:
      serviceAccount: "{{secret('GCP_CREDS')}}"
      projectId: "{{kv('GCP_PROJECT_ID')}}"
      location: "{{kv('GCP_LOCATION')}}"
      bucket: "{{kv('GCP_BUCKET_NAME')}}"
```

Although the configuration was syntactically correct, the environment mapping did not align with how the secrets were defined, which prevented successful authentication.
